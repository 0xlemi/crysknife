# Design Document: Crysknife

## Overview

Crysknife is a Go CLI tool (`crys`) that orchestrates multiple kiro-cli agents in tmux. It manages agent lifecycle, work assignment with workflow tiers, monitoring with auto-nudge/restart, merge queue management, and feature-level tracking. All state lives in `.crysknife/state.json`. Agents run in isolated Git worktrees and interact with the orchestrator through `crys` CLI commands via `execute_bash`.

The binary uses Cobra for CLI routing. Templates and hook scripts are embedded in the binary via Go's `embed` package and copied to the project on `crys init`. Per-worker kiro-cli agent configs are generated programmatically by `crys start` and regenerated by `crys sling` when work is reassigned.

### Key Design Decisions

1. **Cobra for CLI routing** -- each command is a separate file in `cmd/`. Cobra handles flags, help text, and subcommands. No custom argument parsing.

2. **Atomic state writes** -- all writes to state.json go through a single `state.Save()` function that writes to a temp file and renames. This prevents partial writes from concurrent hook invocations.

3. **Embedded templates** -- default templates and hook scripts are embedded in the binary via Go's `embed` package. `crys init` copies them out. The binary is self-contained with no external file dependencies.

4. **Worker configs are generated programmatically** -- `crys start` and `crys sling` build worker JSON configs in Go code (not text/template). This avoids JSON escaping issues and makes per-worker field customization straightforward.

5. **Worktree directory is sibling to project** -- worktrees go in `../<project>-worktrees/worker-1/` etc. This keeps them out of the project directory and avoids confusing editors and tools. The merger operates in the main project directory (no worktree) since it merges to main. The mayor gets a worktree on the main branch.

6. **tmux layout is deterministic** -- window and pane positions are fixed. Mayor is always window 1, dashboard window 2, workers in pairs starting at window 3, merger after workers, then lazygit, nvim, terminal. Pane targets are stored in state.json.

7. **No MCP server** -- agents run `crys` commands via `execute_bash` instead of calling MCP tools. Simpler, more reliable, no per-agent processes. Role enforcement comes from prompts and the command guard hook. MCP can be added later if needed.

8. **Two preToolUse guards** -- `enforce-area.sh` blocks file writes outside the worker's area. `enforce-commands.sh` blocks dangerous bash commands. Both use exit code 2 to block and return error messages via stderr.

9. **`crys stop` does not stop `crys watch`** -- stop only kills agent panes (mayor, workers, merger), not the dashboard window where watch runs. The watch loop will see all agents as "stopped" and idle harmlessly. You Ctrl+C it manually. This matches the "no daemon" design — watch is a foreground process you control.

## Architecture

```
crys (binary)
  |
  |-- cmd/
  |   |-- root.go         root command, binary name "crys"
  |   |-- init.go         creates directories, copies templates, writes base configs
  |   |-- start.go        generates worker configs, worktrees, tmux, launches agents
  |   |-- stop.go         kills panes, updates state
  |   |-- status.go       reads state, prints table
  |   |-- sling.go        renders template, generates task file, regenerates config, nudges
  |   |-- queue.go        add/remove/list tasks in state
  |   |-- watch.go        monitoring loop with heartbeat + pane diffing
  |   |-- nudge.go        manual nudge via tmux send-keys
  |   |-- done.go         mark worker done, add to merge queue, nudge mayor+merger
  |   |-- convoy.go       feature-level grouping
  |   |-- heartbeat.go    update last_activity timestamp
  |   |-- mytask.go       print task summary to stdout
  |   |-- mergequeue.go   display pending merges
  |   |-- mergedone.go    mark merge complete or failed
  |
  |-- internal/
  |   |-- state/state.go      state.json read/write with atomic saves
  |   |-- tmux/tmux.go        tmux session/window/pane management
  |   |-- agentcfg/agentcfg.go  kiro-cli agent config generation
  |   |-- template/template.go  workflow tier template rendering
  |   |-- monitor/monitor.go    heartbeat + pane diffing detection
  |
  |-- templates/           embedded default templates
  |-- hooks/               embedded hook scripts
  |-- main.go              entry point
  |-- go.mod, go.sum
```

### Data Flow

```
crys init
  writes: .crysknife/ (state.json, templates/, hooks/)
  writes: .kiro/agents/ (mayor.json, merger.json)
  writes: .kiro/specs/ (tasks/, plan.md, principles.md)

crys start --workers 4
  reads:    .crysknife/state.json, templates/
  generates: .kiro/agents/worker-1..4.json
  runs:     git worktree add (per worker)
  runs:     ln -s (symlinks per worktree)
  runs:     tmux new-session, new-window, split-window
  runs:     tmux send-keys "kiro-cli chat --agent <name>" (per pane)
  writes:   .crysknife/state.json (all agents registered)

crys sling worker-1 --task "Add auth" --tier full --area "src/auth/"
  reads:    .crysknife/templates/full.md
  writes:   .kiro/specs/tasks/worker-1.md (rendered template)
  generates: .kiro/agents/worker-1.json (updated area restrictions)
  writes:   .crysknife/state.json (worker status, task, branch, area, tier)
  runs:     tmux send-keys (nudge worker)

crys done worker-1
  writes:   .crysknife/state.json (status -> "done", branch -> merge queue)
  runs:     tmux send-keys (nudge mayor + merger)

crys watch (polling loop)
  reads:    .crysknife/state.json (heartbeat timestamps)
  reads:    tmux capture-pane (fallback: pane content diffing)
  runs:     tmux send-keys (auto-nudge idle agents)
  runs:     tmux kill-pane + kiro-cli chat --agent (auto-restart dead agents)
  writes:   .crysknife/state.json (status updates)
  prints:   dashboard to stdout

crys merge-done <branch>
  writes:   .crysknife/state.json (remove from merge queue, update worker status)
  runs:     tmux send-keys (nudge mayor)
```

## Data Models

### State File Schema (`.crysknife/state.json`)

Initial state created by `crys init`:

```json
{
  "project": "<directory-name>",
  "created": "<UTC-timestamp>",
  "agents": [],
  "queue": [],
  "convoys": [],
  "merge_queue": []
}
```

After `crys start --workers 4`, the agents array contains 6 entries (1 mayor + 4 workers + 1 merger). See `Agent` struct for field definitions.

Valid agent statuses: `"idle"`, `"working"`, `"done"`, `"merged"`, `"merge-failed"`, `"nudged"`, `"stopped"`, `"waiting"`.

Note: after auto-restart by `crys watch`, the agent status is set to `"working"` (not a separate "restarted" status).

Valid merge queue statuses: `"ready"`.

### Workflow Tier Templates

Three templates in `.crysknife/templates/`:

- `full.md` -- requirements review, design, implement, self-review, unit tests, full test suite, summary
- `standard.md` -- implement, unit tests, test suite, summary
- `quick.md` -- implement, verify, done

All templates use `{{AGENT_ID}}`, `{{TASK_NAME}}`, `{{AREA}}`, `{{BRANCH}}` as variables.

### Agent Config Schema

See `AgentConfig` struct. Three base configs:

- `mayor.json` -- fs_read, fs_write, execute_bash, grep, glob. Hooks: agentSpawn (crys status --json), preToolUse (enforce-commands.sh), stop (crys heartbeat).
- `worker-N.json` (generated) -- adds code tool. Hooks: agentSpawn (crys my-task), preToolUse (enforce-area.sh + enforce-commands.sh), stop (crys heartbeat). toolsSettings restricts fs_write paths.
- `merger.json` -- same tools as mayor. Hooks: agentSpawn (crys merge-queue --json), preToolUse (enforce-commands.sh with merger allowances), stop (crys heartbeat).

## Correctness Properties

### Property 1: Atomic state writes

*For any* sequence of concurrent `crys` commands writing to State_File, the file SHALL always contain valid JSON after each write completes.

**Validates: Requirements 15.1**

### Property 2: Init idempotency guard

*For any* project where `.crysknife/` already exists, `crys init` SHALL exit with a non-zero code and leave existing files unchanged.

**Validates: Requirements 1.7**

### Property 3: Agent count consistency

*For any* `crys start --workers N`, the State_File SHALL contain exactly N+2 agents (N workers + 1 mayor + 1 merger) after startup completes.

**Validates: Requirements 2.6, 2.7, 2.8**

### Property 4: Heartbeat timestamp freshness

*For any* `crys heartbeat <id>` call, the agent's `last_activity` in State_File SHALL be within 1 second of the current UTC time.

**Validates: Requirements 12.1**

### Property 5: Worktree isolation

*For any* worker started by `crys start`, the worker's Git worktree SHALL be on a unique branch and in a separate directory from all other workers.

**Validates: Requirements 2.2**

### Property 6: Symlink validity

*For any* worker worktree created by `crys start`, the symlinks for `.crysknife/` and `.kiro/specs/` SHALL resolve to the main project's real directories.

**Validates: Requirements 2.3**

### Property 7: Stop updates state

*For any* `crys stop` call, all targeted agents in State_File SHALL have status "stopped" after the command completes.

**Validates: Requirements 3.2, 3.3**

### Property 8: Task context completeness

*For any* agent with a current task in State_File, `crys my-task <id>` SHALL output all non-null fields (id, role, status, task, branch, area, task_file).

**Validates: Requirements 13.1**

### Property 9: Sling state consistency

*For any* `crys sling` call, the worker's State_File entry, Task_File, and Agent_Config SHALL all reflect the same task, area, and branch.

**Validates: Requirements 5.2, 5.3, 5.5, 16.6**

### Property 10: Done triggers merge queue

*For any* `crys done <worker>` call, the worker's branch SHALL appear in the Merge_Queue with status "ready".

**Validates: Requirements 9.1, 9.2**

### Property 11: Merge-done cleans queue

*For any* `crys merge-done <branch>` call, the branch SHALL no longer appear in the Merge_Queue.

**Validates: Requirements 11.1**

### Property 12: Area enforcement

*For any* `fs_write` call by a worker to a path outside its assigned area, the enforce-area.sh hook SHALL block the operation (exit code 2).

**Validates: Requirements 17.1, 17.2**

### Property 13: Command guard

*For any* `execute_bash` call matching a blocked pattern, the enforce-commands.sh hook SHALL block the operation (exit code 2).

**Validates: Requirements 17.3, 17.4**

### Property 14: Watch detection accuracy

*For any* agent whose last_activity exceeds the nudge threshold, `crys watch` SHALL classify it as Idle and trigger a nudge (unless --no-nudge).

**Validates: Requirements 7.3**

## Error Handling

| Scenario | Behavior |
|---|---|
| `crys init` when `.crysknife/` exists | Exit code 1, "Crysknife already initialized in this directory" |
| `crys start` when tmux session exists | Exit code 1, "tmux session '<name>' already exists. Run crys stop first." |
| `crys start` when `.crysknife/` missing | Exit code 1, "Not a Crysknife project. Run crys init first." |
| `crys sling` with unknown worker-id | Exit code 1, "Agent '<id>' not found in state.json" |
| `crys sling` with invalid tier | Exit code 1, "Unknown tier '<tier>'. Use full, standard, or quick." |
| `crys sling --from-queue` with empty queue | Exit code 1, "Work queue is empty" |
| `crys done` with unknown worker-id | Exit code 1, "Agent '<id>' not found in state.json" |
| `crys done` when worker has no branch | Exit code 1, "Worker '<id>' has no branch assigned" |
| `crys merge-done` with unknown branch | Exit code 1, "Branch '<branch>' not found in merge queue" |
| `crys heartbeat` with unknown agent-id | Exit code 1, error to stderr |
| `crys my-task` with unknown agent-id | Exit code 1, error to stderr |
| `crys queue remove` with unknown task-id | Exit code 1, "Task '<id>' not found in queue" |
| state.json missing on read | Return empty default state (no error) |
| state.json contains invalid JSON | Exit code 1, "Corrupted state.json: <parse error>" |
| `git worktree add` fails | Exit code 1, print git error, clean up partial worktrees |
| tmux not installed | Exit code 1, "tmux is required but not found in PATH" |
| kiro-cli not installed | Exit code 1, "kiro-cli is required but not found in PATH" |
| Template file missing | Exit code 1, "Template '<tier>.md' not found in .crysknife/templates/" |

## Testing Strategy

- State read/write: unit tests with temp directories. Test atomic writes, concurrent access, empty state handling, corrupt JSON handling.
- Agent config generation: unit tests verifying JSON output matches expected structure for each role.
- Template rendering: unit tests verifying variable substitution for each tier.
- tmux operations: integration tests requiring tmux (skip with `testing.Short()`).
- Hook scripts: shell tests verifying exit codes for allowed/blocked paths and commands.
- Monitor logic: unit tests for heartbeat threshold calculations and state classification.
- CLI commands: integration tests for the full init -> start -> sling -> done -> stop flow.
- Run: `go test ./...`
